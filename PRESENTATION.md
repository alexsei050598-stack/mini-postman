# Mini Postman QA Tool

## Слайд 1. Проект
**Mini Postman** — веб-инструмент для ручного тестирования API.

Цель проекта:
- ускорить проверку API без тяжёлых desktop-приложений;
- добавить QA-функции, которых обычно нет в простых учебных клиентах;
- сделать наглядную демо-версию для портфолио.

Кому полезно:
- QA-инженерам;
- junior backend/frontend разработчикам;
- стажёрам для быстрой проверки endpoint'ов.

---

## Слайд 2. Проблема
Что неудобно в обычных мини-клиентах:
- только GET/POST;
- слабая обработка ошибок;
- нет истории запросов;
- не видно качества ответа с первого взгляда;
- сложно воспроизводить сетевые кейсы (slow/offline).

---

## Слайд 3. Что реализовано
Функциональность:
- HTTP-методы: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, `OPTIONS`;
- кастомные headers;
- body для методов с телом;
- валидация URL/body/headers до отправки;
- история запросов в `localStorage`;
- цветной статус-бейдж (2xx/4xx/5xx);
- QA-режимы: `Normal`, `Slow 3G`, `Offline`;
- copy response + clear формы;
- анализ типовых ошибок (`403`, `404`, `5xx`).

---

## Слайд 4. Архитектура (простая)
Технологии:
- `index.html` — структура UI;
- `style.css` — дизайн и визуальная иерархия;
- `script.js` — логика запросов и обработки ответов;
- `localStorage` — хранение истории.

Поток:
1. Пользователь заполняет форму.
2. Скрипт валидирует данные.
3. Отправка через `fetch()`.
4. Парсинг ответа по `content-type`.
5. Вывод статуса, анализа и тела ответа.
6. Сохранение в историю.

---

## Слайд 5. Качество и устойчивость
Что сделано для стабильности:
- корректная обработка пустых ответов (`HEAD`, `204`, `304`);
- fallback для текстовых и бинарных ответов;
- отдельные сообщения для сетевых ошибок;
- офлайн-эмуляция без падения интерфейса;
- понятные ошибки в UI вместо stack trace.

Итог:
- инструмент не "ломается" при реальных кейсах;
- быстро понятно, проблема в клиенте, сети или API.

---

## Слайд 6. Demo-сценарий (2-3 минуты)
1. `GET /posts/1` -> показать `200` и JSON.
2. `POST /posts` -> показать `201`.
3. `HEAD /posts/1` -> показать `200` и `[no content]`.
4. `GET /status/404` -> показать подсказку про endpoint.
5. Включить `Slow 3G` и `Offline` -> показать QA-режимы.
6. Клик по истории -> автоподстановка формы.

---

## Слайд 7. Ценность для работодателя
Этот проект показывает:
- умение делать не только "работает", но и "удобно тестировать";
- QA-мышление: edge cases, сообщения об ошибках, воспроизводимость;
- продуктовый подход: UX, демо-готовность, понятный README.

Что можно добавить дальше:
- вкладки запросов как в Postman;
- env variables (`{{baseUrl}}`, `{{token}}`);
- экспорт/импорт коллекций;
- автотесты (Playwright/Vitest) для ключевых сценариев.

---

## Короткий спич (30-40 секунд)
Я сделал Mini Postman как QA-ориентированный API клиент. 
В отличие от базовых примеров, он поддерживает все основные HTTP-методы, историю запросов, обработку ошибок и эмуляцию сети через Slow 3G и Offline. 
Инструмент устойчив к edge-case сценариям, например `HEAD` или пустым ответам, и сразу даёт понятный визуальный статус. 
Проект демонстрирует не только JavaScript-навыки, но и практический подход к качеству и удобству тестирования API.
